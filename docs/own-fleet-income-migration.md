# Миграция дохода собственного транспорта на архитектуру "заявка → рейс"

Дата: 2026-07-22. Пять этапов, коммиты `9815d23` (Этап 0) → `8635f33`/`3faa7b4` (фиксы
пробега/дохода, предшествовавшие расследованию) → `8cb7586` (Этап 2) → `4c058c1` (Этап 3) →
текущий (Этап 4).

## Зачем

Расследование показало: доход по собственным машинам считался в 6 разных местах
6 разными способами (сопоставление заявок с рейсом по датам в одних местах, прямая сумма
заявок по типу — в других), из-за чего цифры расходились между карточкой рейса,
аналитикой, «Финансовым директором», Dashboard и отчётами. Причина — у заявки не было
явной, обязательной связи с рейсом машины: принадлежность определялась тем, попадает ли
`Trip.tripDate` в диапазон `[VehicleTrip.departureDate, returnDate]`, что ненадёжно (разрывы
между рейсами, рейсы без `returnDate`, задвоение при пересекающихся окнах).

## Итоговая архитектура

Единственный источник принадлежности заявки рейсу — явное поле `Trip.vehicleTripId`
(уже существовало в схеме, просто не заполнялось при жизни рейса, только при закрытии).
Даты (`departureDate`, `returnDate`, `tripDate`) больше НЕ участвуют в определении дохода —
только для отображения, поиска, сортировки, подсказок и отчётов по периодам.

```
Заявка (own_transport, машина назначена)
  └─ Открытых рейсов машины: 0 → ждёт привязки | 1 → привязывается автоматически | >1 → выбор диспетчера
Рейс создаётся ТОЛЬКО диспетчером вручную (даты обязательны, как и раньше) — система рейс не создаёт
Доход рейса  = SUM(заявки, vehicleTripId = этот рейс)
Доход машины = SUM(доход её рейсов)          — заявки без vehicleTripId в доход НЕ входят
Доход парка  = SUM(доход всех машин)
```

Заявка без рейса — это не ошибка, а нормальное переходное состояние («в работе»), пока
диспетчер не заведёт/выберет подходящий рейс. Она видна в счётчике/разделе «Ожидают
привязки», не пропадает молча.

## Переключённые потребители (Этап 3)

`GET /api/vehicle-trips/[id]`, `GET /api/vehicles/[id]/economics`, `GET /api/vehicle-analytics`,
`GET /api/director-finance` (`ownTransport`), `GET /api/dashboard` (`ownFleet`) — все теперь
читают доход через `Trip.vehicleTripId`, не через сопоставление по датам.

**Намеренно НЕ тронут**: `GET /api/reports/trips` — он никогда не сопоставлял заявки с
рейсом по датам, это прямая сумма заявок по типу за период (другое измерение отчёта —
«что произошло в эту дату», а не «доход рейса») — присоединять его к
`vehicleTripId`-группировке было бы неверно по смыслу.

Раздел «Ежедневный отчёт» (`daily-reports`), который тоже использовал старую логику,
удалён из TMS 23.07.2026 по итогам функционального аудита — его показатели дублировали
`/dashboard` и `/director-finance` (для собственного транспорта — уже по новой
архитектуре), а часть цифр (own_transport в сводке день/неделя) была построена на старой,
рассинхронизированной логике.

## Что устарело / можно удалить в будущем

- `lib/vehicle-trips/revenue.ts`: `findMatchingTrips`, `matchTripsInRange`, `resolveMatchRangeEnd` —
  используются ТОЛЬКО в `lib/finance/income-diagnostic.ts` для сверки. Если диагностика
  больше не нужна — можно удалить оба файла целиком.
- `computeVehicleTripFinancials` (там же) — всё ещё используется в
  `app/api/vehicle-trips/[id]/route.ts`, удалять не нужно.
- `Setting.income_calc_mode` — можно удалить, если решите, что историческая метка не нужна;
  ни на что в рантайме не влияет.

## Найденный по пути и исправленный баг

`POST /api/vehicle-trips/[id]/recalculate-income` (пересчёт дохода закрытого рейса после
правки дат) при старой логике **пересобирал состав заявок заново по датам**, включая
отвязку уже привязанных — прямое нарушение правила «состав закрытого рейса зафиксирован».
Исправлено на Этапе 4: эндпоинт больше не трогает привязку, только пересчитывает сумму по
уже привязанному набору.

## Разовая миграция данных (Этап 1)

Для всех рейсов, кроме уже закрытых (там `vehicleTripId` был проставлен при закрытии и
раньше), проставлен `vehicleTripId` по уже проверенной date-matching логике — 24 заявки.
14 заявок не покрывались ни одним рейсом ни на одну дату («сироты», в основном исторические,
до того как рейсы стали вестись системно) — остались `vehicleTripId = null`, видны в разделе
«Ожидают привязки».

## Видимый эффект для пользователей после деплоя

Агрегированный «доход по собственному транспорту» на `/director-finance` и Dashboard упадёт
с ≈78,9 млн ֏ (считалось по всем заявкам напрямую) до ≈50,4 млн ֏ (только явно привязанные) —
это ожидаемо и обсуждено с пользователем заранее, не регрессия.
